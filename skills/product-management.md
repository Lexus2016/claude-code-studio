## Core Frameworks

**Jobs-to-Be-Done**: Define features by the job users are trying to accomplish, not the solution they're requesting. "I want drag-and-drop" = want; "I need to reorganize my work quickly" = job. Design for the job.

**RICE Prioritization**: Score features by Reach (users/quarter) Ã— Impact (0.25â€“3) Ã— Confidence (%) Ã· Effort (person-weeks). Removes HiPPO effect; forces explicit assumptions.

**Working Backwards (Amazon)**: Write the press release and FAQ before building. If you can't write a compelling "customer benefits" section, the feature isn't ready to build. Forces clarity on value proposition.

**Shape Up (Basecamp)**: Fixed 6-week cycles, variable scope. Define appetite (time budget) before defining solution. "How much is this worth to us?" before "How long will it take?"

## Key Practices

**Opportunity Assessment**: Before any feature, answer: What problem is this solving? Who has this problem? How do we know? What's the alternative if we don't build this? What does success look like and how do we measure it?

**Continuous Discovery**: Weekly user interviews, not quarterly research sprints. Talk to 2-3 users every week. Build a habit, not a project.

**Outcome-Based Roadmap**: Organize by problems to solve and metrics to move, not features to ship. "Reduce checkout abandonment by 20%" beats "Add Apple Pay" on a roadmap.

## Anti-Patterns to Avoid

- **Feature Factory**: Teams measure output (features shipped) not outcomes (problems solved). Roadmap becomes a feature request list. Success = shipping, not impact.

- **HiPPO-Driven Decisions**: Highest Paid Person's Opinion drives priorities without user evidence. Leads to features nobody uses and engineers who've lost faith in the product process.

- **Scope Creep Acceptance**: "While we're at it..." is how ship dates slip and complexity explodes. Fixed appetite cycles (Shape Up) or hard scope cuts are the only cure.

- **Discovery After Commitment**: Building the solution before validating the problem. Sunk cost bias makes it impossible to kill. Validate problem first, solution second.
